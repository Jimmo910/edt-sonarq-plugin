Плагин для **1C:Enterprise Development Tools (EDT)**, который показывает замечания
**SonarQube** по BSL-коду прямо в IDE. Основной режим читает результаты анализа с сервера
SonarQube через Web API и учитывает ветки там, где это поддерживает редакция сервера
(коммерческие редакции); при этом плагин работает и с SonarQube Community Edition
(без веток). Дополнительно доступен локальный режим без сервера — он запускает
**BSL Language Server** прямо по исходникам проекта.

## Возможности

- **Окно «Замечания SonarQube»** — дерево замечаний активного проекта с группировкой по
  файлу или по правилу, фильтрами по критичности и типу и текстовым поиском.
- **Переход к коду** — двойной клик по замечанию открывает модуль на нужной строке;
  одиночный клик показывает описание правила в нижней панели.
- **Автоопределение git-ветки** — окно определяет текущую ветку проекта и запрашивает
  замечания именно по ней (на коммерческих редакциях сервера).
- **Привязка к проекту** — каждый проект связывается с ключом проекта SonarQube, с
  необязательной фиксированной веткой и префиксом пути в репозитории.
- **Замечания прямо в коде** — транзиентные маркеры в стандартном окне Problems и
  подчёркивания в редакторе.
- **Запуск анализа из окна** — по кнопке (локальный scanner или CI-триггер).
- **Локальный режим без сервера** — анализ BSL Language Server прямо по исходникам, без
  SonarQube и без установки Java.
- **Настройка применяемых проверок** — страница «Проверки BSL» с группировкой по типу или
  тегу (таксономия самого BSL Language Server) и кнопкой «Применить рекомендованный профиль».
- **Область анализа локального режима** — на проект: базовая ветка (показывать замечания
  только на изменённых относительно неё строках) и фильтр подсистем.
- **Быстрое подавление ложных срабатываний** — правый клик в окне замечаний или Quick Fix в
  окне Problems сам вписывает подавляющие комментарии BSL Language Server вокруг строки.

## Требования

- 1C:Enterprise Development Tools **2026.1** (target-платформа сборки).
- Java 17 — та, на которой уже работает EDT (отдельно устанавливать не нужно).
- Для режима сервера — сервер SonarQube с уже проанализированным BSL-проектом (например,
  анализ на CI с плагином
  [sonar-bsl-plugin-community](https://github.com/1c-syntax/sonar-bsl-plugin-community)).
- Для локального режима — доступ в интернет один раз, при первом анализе, чтобы автоматически
  скачать нативную сборку BSL Language Server (если не указан локальный файл).

## Установка

**Из p2-архива обновлений** (рекомендуется):

1. Скачайте zip-архив обновлений `ru.jimmo.edt.sonarq.repository` со страницы **Releases**
   проекта (или соберите сами — см. [Сборка из исходников](#сборка-из-исходников)).
2. В EDT: `Help` → `Install New Software…` → `Add…` → `Archive…`, укажите скачанный zip,
   выберите функцию **SonarQ in EDT** и пройдите мастер до конца.
3. Перезапустите EDT по запросу.

**Из командной строки** (p2 director) — для скриптовой/автономной установки:

```
<каталог-EDT>/1cedtc.exe -nosplash -application org.eclipse.equinox.p2.director ^
  -repository file:/<путь-к-распакованному-репозиторию> ^
  -installIU ru.jimmo.edt.sonarq.feature.feature.group ^
  -vm <путь-к-jdk17>/bin/javaw.exe
```

## Как открыть окно

**Окно** → **Показать панель** → **Другие…** → категория **SonarQube** → **Замечания SonarQube**
(в английской локализации EDT: `Window` → `Show View` → `Other…` → `SonarQube` → `SonarQube Issues`).


## Окно замечаний

В верхней части окна расположен тулбар, ниже — дерево замечаний, под ним — панель описания
правила, а в самом низу — строка статуса. Кнопки тулбара показывают иконки, а не текстовые
подписи — название и назначение кнопки видны во всплывающей подсказке при наведении.

| Элемент | Что делает |
|---|---|
| **Обновить** | Перечитать замечания. В режиме сервера — забрать их с сервера; в локальном режиме — запустить локальный анализ заново. |
| **Запустить анализ ветки** | В режиме сервера — проанализировать текущее состояние рабочей копии настроенным способом (локальный scanner или CI-триггер). При попытке перезаписать уже проанализированную ветку показывается диалог подтверждения. В локальном режиме действует так же, как **Обновить**. |
| **Проект** | Выбор проекта воркспейса, если их несколько. |
| **Критичность** | Мультивыбор критичностей: BLOCKER, CRITICAL, MAJOR, MINOR, INFO. |
| **Тип** | Фильтр по типу: BUG, VULNERABILITY, CODE_SMELL. |
| Поле **«Фильтр по правилу или сообщению»** | Текстовый фильтр по ключу правила и тексту сообщения. |
| **Группировать по файлу** / **по правилу** / **по критичности** | Переключатель структуры дерева. Колонка, по которой идёт группировка, автоматически скрывается (не дублирует заголовки групп). |

Дополнительно в окне:

- **Строка статуса** внизу показывает число замечаний, ветку и время последнего обновления, а
  также — сколько из загруженных замечаний **не показаны в окне Problems**: путь замечания не
  сопоставляется ни с одним файлом проекта, либо сопоставляется, но файла всё равно нет на диске
  даже после обновления структуры проекта; это часть ценности окна замечаний поверх стандартного
  Problems, где такие замечания не видны вовсе.
- При сбое строка статуса показывает краткий заголовок ошибки (например, ошибку аутентификации
  или сети); полный текст ошибки виден во всплывающей подсказке при наведении на строку, а
  кнопка **«Подробнее»** рядом открывает его целиком в диалоге. Ошибка нехватки памяти в
  локальном режиме прямо указывает на настройку «Максимум памяти BSL LS» (см. [«Настройка
  локального режима»](#настройка-локального-режима-без-сервера) выше).
- В локальном режиме строка статуса показывает свой собственный прогресс — отдельное название
  фонового задания и фазы (в том числе скачивание движка BSL Language Server при первом
  запуске, ~170 МБ) — вместо серверной формулировки про скачивание; при каждом новом запуске
  (**Обновить** или смена проекта) текст статуса и ошибки очищаются заново, не показывая
  устаревший результат прошлого запуска.
- **Двойной клик** по замечанию открывает соответствующий модуль на нужной строке;
  **одиночный клик** загружает описание правила в нижнюю панель.
- Замечания, для которых **файл не найден локально**, показываются серым; тултип поясняет,
  что переход недоступен.
- Если текущая git-ветка ещё не анализировалась на сервере, над деревом появляется плашка
  **«Ветка … ещё не анализировалась на сервере»** со ссылкой **«Отправить ветку на анализ»**
  (только в режиме сервера, на коммерческих редакциях SonarQube).

## Настройка режима сервера

### 1. Подключение к серверу

**Настройки** → **SonarQube**:

- **Режим** — выберите **«Сервер SonarQube»**.
- **URL сервера** — адрес сервера SonarQube.
- **Токен** — пользовательский токен. Как создать: в веб-интерфейсе SonarQube откройте
  **My Account → Security → Generate Tokens** и создайте токен типа **User Token**.
  Для кнопки запуска анализа токену дополнительно нужно право **Execute Analysis**.
  Токен хранится в **Eclipse Secure Storage**, а не в обычных настройках.
- **Тайм-аут, сек.** — тайм-аут HTTP-запросов.
- Кнопка **«Проверить соединение»** проверяет URL и токен и показывает версию сервера.


### 2. Привязка проекта

**Правый клик по проекту** → **Свойства** → **SonarQube**:

- **Имя проекта** — ключ проекта (project key) в SonarQube. Кнопка **«Заполнить»**
  подбирает его по имени проекта EDT (подсказка выводится прямо в поле).
- **Фиксированная ветка** — если оставить пустой, ветка определяется автоматически из git.
- **Префикс пути в репозитории** — заполняется только если проект EDT лежит в подкаталоге
  git-репозитория, который анализирует CI; иначе оставьте пустым.

### 3. Загрузка замечаний

Откройте окно **«Замечания SonarQube»** и нажмите **Обновить**.

### 4. Запуск анализа (необязательно)

Группа **«Запуск анализа»** на странице настроек задаёт, как работает кнопка
**«Запустить анализ ветки»**. **Способ запуска**:

- **Локальный scanner (скачать автоматически)** — плагин сам скачает sonar-scanner CLI.
- **Локальный scanner (указанный путь)** — использовать scanner по заданному **Пути к scanner**.
- **CI-триггер (URL)** — POST-запрос на webhook CI. Задаётся **URL CI-триггера** с
  плейсхолдером `{branch}` (подставляется текущая ветка) и необязательный **Секрет CI**
  (отправляется как заголовок `Authorization`, хранится в Secure Storage).

Поле **«Доп. параметры scanner»** добавляет аргументы к обоим scanner-режимам.

Пример URL для GitLab pipeline trigger:

```
https://gitlab.example.com/api/v4/projects/<id>/trigger/pipeline?token=<trigger_token>&ref={branch}
```

> **Предупреждение.** GitLab ожидает trigger-токен в query-параметре URL, поэтому его
> вписывают прямо в шаблон URL — и тогда он хранится в **обычных (незашифрованных)**
> настройках. Используйте маловластный trigger token.

### 5. Ветки

На коммерческих редакциях SonarQube окно показывает ветку, соответствующую текущей git-ветке
проекта. На SonarQube Community Edition ветки не поддерживаются — показывается основная ветка,
а запуск анализа всегда попадает в единственную ветку по умолчанию (перед перезаписью уже
проанализированной ветки выводится диалог подтверждения).

## Настройка локального режима (без сервера)

**Настройки** → **SonarQube** → **Режим** → **«Локальный анализ (BSL Language Server)»**.

- **Источник**:
  - **«Скачать автоматически (из интернета, ~170 МБ однократно)»** — по умолчанию; плагин
    скачает нативную сборку BSL Language Server в свой каталог при первом анализе. Java при
    этом не требуется.
  - **«Указать локальный файл»** — задать **Путь к BSL Language Server** вручную (кнопки
    **«Обзор…»** и **«Проверить»**; **«Проверить»** запускает файл с `--version`).
- **«Максимум памяти BSL LS (ГБ)»** (по умолчанию 4) — максимальный размер кучи JVM, который
  плагин прописывает в скачанный движок перед каждым локальным анализом. Увеличьте значение,
  если анализ падает с нехваткой памяти на большой конфигурации (сообщение об ошибке в окне
  замечаний прямо ссылается на эту настройку). Действует только на управляемый скачанный движок
  — на указанный вручную исполняемый файл (**«Указать локальный файл»**) не влияет, поле в этом
  случае недоступно.
- Раздел **«Движок анализа»** показывает, установлен ли уже скачанный BSL Language Server
  (**«установлен»** / **«не установлен»**), и кнопка **«Удалить скачанный движок»** удаляет его
  — например, чтобы принудительно скачать заново; при следующем локальном анализе он скачается
  автоматически.
- Каждое нажатие **«Обновить»** запускает свежий локальный анализ.
- Какие проверки применяются — настраивается отдельно, см. [«Выбор применяемых
  проверок»](#выбор-применяемых-проверок) ниже; по умолчанию включены все проверки, встроенные
  в BSL Language Server.
- Ветки, кнопка запуска анализа и настройки CI в этом режиме не действуют — проект
  анализируется целиком.

### Выбор применяемых проверок

Список проверок, которые применяет локальный анализ, настраивается на отдельной странице
**Настройки** → **SonarQube** → **Проверки BSL** (класс `BslChecksPreferencePage`).

- Проверки показаны деревом (колонки **Ключ** и **Название**), структуру которого задаёт
  выпадающий список **«Группировать по»** над деревом: **«По типу»** (значение по умолчанию) —
  группы по собственному типу проверки в BSL Language Server (Code smell, Error, Vulnerability,
  Security Hotspot); **«По тегу»** — группы по тегам BSL Language Server (`standard`,
  `badpractice`, `performance`, `sql`, `brainoverload` и т. п.; проверка с несколькими тегами
  показывается в каждой своей группе); **«Без группировки»** — плоский список. Тип и теги
  каждой проверки также видны в тултипе её строки. У каждой группы — свой флажок (переключает
  разом все проверки внутри неё; показывает промежуточное состояние, если часть проверок
  включена, а часть выключена) и собственный счётчик **«Выключено N из M»**. Включённая
  проверка (флажок установлен) применяется, выключенная — игнорируется. По умолчанию включены
  все проверки.
- Кнопка **«Применить рекомендованный профиль»** и тултип «Дублирует проверку EDT: …» не
  зависят от выбранной группировки дерева — они используют отдельную, встроенную в плагин
  классификацию причин (дубли валидатора EDT, требующие настройки под проект, неуместные в EDT,
  прочие) и одним нажатием выключают рекомендованный набор — дубли валидатора EDT + проверки,
  требующие настройки под проект, + неуместные в EDT (категорию «прочие» кнопка не трогает).
  Это осознанное действие пользователя (opt-in): пока кнопка не нажата, ничего не меняется —
  включены все проверки, как и раньше.
- Эта классификация причин — первый, заведомо неполный срез, встроенный в плагин; сообщество
  может уточнять его через Pull Request (см.
  [issue #3](https://github.com/Jimmo910/edt-sonarq-plugin/issues/3)).
- Поле фильтра ищет по ключу или названию; кнопки **«Включить все»** / **«Выключить все»**
  переключают сразу все проверки независимо от текущего фильтра и категории. Строка
  **«Выключено N из M»** под деревом показывает общее число отключённых проверок.
- Полный каталог из 186 известных проверок встроен в плагин, поэтому дерево показывает все
  категории сразу же, даже до первого локального анализа или нажатия кнопки получения списка.
  Встроенный каталог дополняется (но не заменяется) кэшем, который заполняется автоматически
  после каждого успешного локального анализа, либо вручную — кнопкой **«Получить список
  проверок»**: она запускает анализ пустой временной папки, чтобы получить полный перечень
  проверок без единого реального замечания. Если BSL Language Server ещё не скачан, первое
  использование кнопки (как и первый локальный анализ) скачает его — это около 170 МБ и требует
  доступа в интернет.
- Выбор с этой страницы — отмеченный вручную или кнопкой «Применить рекомендованный профиль» —
  применяется и хранится так же, как и раньше: через генерируемый конфигурационный файл BSL
  Language Server, который передаётся анализу флагом `--configuration`. Если в проекте уже есть
  свой файл `.bsl-language-server.json` (в корне проекта или в каталоге `src/`), он всегда имеет
  приоритет: анализ использует его вместо сгенерированного, а выбор на странице настроек для
  такого проекта игнорируется.

### Область анализа

На странице **Свойства проекта → SonarQube** (см. [«Привязка проекта»](#2-привязка-проекта)
выше) есть группа **«Область анализа (локальный режим)»** с двумя независимыми фильтрами. Оба
действуют только в локальном режиме анализа и по умолчанию пустые — это значит «анализировать
всё», как и раньше (без потери обратной совместимости).

- **«Базовая ветка»** — редактируемый выпадающий список, предзаполненный именами локальных
  git-веток проекта. Если значение задано, показываются только замечания на строках, изменённых
  относительно этой базы (базовая ветка или коммит в сравнении с рабочей копией, включая
  незакоммиченные правки) — та же идея, что и скрытие ошибок базовой ветки в самой EDT. Если
  значение не удаётся разрешить (проект вне git-репозитория или указана несуществующая ветка/
  коммит), фильтр отключается и показываются все замечания — без молчаливого скрытия.
- **«Подсистемы»** — дерево с флажками, прочитанное из каталога `src/Subsystems` проекта
  (вложенные подсистемы читаются рекурсивно). Пустой выбор (ни один флажок не отмечен) —
  анализируется весь проект; отметьте одну или несколько подсистем, чтобы ограничить анализ
  ими — отметка подсистемы включает и все вложенные в неё подсистемы (это делает сам BSL Language
  Server через параметр диагностики `subsystemsFilter`). Кнопка **«Обновить»** перечитывает
  дерево с диска (например, после добавления новой подсистемы), не сбрасывая текущий выбор.

Если в проекте уже есть свой файл `.bsl-language-server.json`, он имеет приоритет над
сгенерированной конфигурацией проверок, и вместе с ней — как и выбор отключённых проверок — для
такого проекта не применяется и фильтр подсистем. А построчный фильтр по базовой ветке от
конфигурации проверок не зависит и продолжает применяться и в этом случае.

## Маркеры в редакторе

Группа **«Маркеры в редакторе»** на странице настроек:

- **Показывать замечания в стандартной панели «Проблемы»** (включено по умолчанию) —
  замечания попадают в стандартное окно **Eclipse Problems** («Проблемы») и отображаются
  подчёркиваниями в редакторе. Соответствие критичности: BLOCKER/CRITICAL → ошибки, MAJOR →
  предупреждения, остальное → информация. Выключите флажок — и замечания будут видны только в
  собственном окне плагина «Замечания SonarQube».
- **«Автоматически обновлять в фоне»** + **«Интервал, мин.»** — периодическое фоновое
  обновление замечаний и маркеров даже при закрытом окне (только в режиме сервера).

> **Важно.** Эти маркеры **не показываются** в собственной панели EDT **«Ошибки
> конфигурации»** — она отображает только замечания встроенного валидатора модели EDT, а
> сторонние плагины не умеют добавлять в неё свои записи. Замечания SonarQube видны именно в
> стандартной панели **Problems** («Проблемы») и подчёркиваниями в редакторе, как описано выше.

Маркеры **транзиентные**: они пересоздаются при каждом обновлении, а после перезапуска EDT
окно Problems пусто до первого обновления.

## Быстрое подавление ложных срабатываний

Если конкретное замечание — ложное срабатывание, его можно подавить прямо из EDT, без ручной
правки кода:

- **правый клик** по замечанию в окне **«Замечания SonarQube»** → **«Подавить замечание»**;
- либо **Quick Fix** на маркере этого замечания в окне Problems.

Действие вставляет вокруг отмеченной строки пару комментариев BSL Language Server:
`// BSLLS:<КлючПравила>-off` строкой выше и `// BSLLS:<КлючПравила>-on` строкой ниже. Эту пару
понимают и сам BSL Language Server (локальный режим), и серверный `sonar-bsl-plugin-community`
(тот же движок диагностик) — оба перестают сообщать об этом правиле на отмеченной строке;
копировать комментарии вручную не нужно.

Если файл уже открыт в редакторе, правка вносится прямо в открытый документ и её можно отменить
(Undo); иначе она сразу сохраняется в файл на диске.

## Как это работает

**Режим сервера.** Плагин выступает читателем Web API SonarQube: он загружает замечания по
ключу проекта и ветке (`/api/issues/search` с пагинацией), описания правил (`/api/rules/show`)
и определяет редакцию сервера, чтобы понять, поддерживаются ли ветки. Дополнительно кнопка
**«Запустить анализ ветки»** может сама запустить анализ — локальным scanner (с
автоскачиванием или по указанному пути) или через CI-триггер — и дождаться, пока сервер
обработает отчёт, после чего окно обновляется.

**Локальный режим.** Плагин запускает нативную сборку BSL Language Server в режиме
`--analyze` по исходникам проекта, получает отчёт в формате **SARIF** и строит из него то же
дерево замечаний, фильтры и маркеры. Сервер SonarQube и Java при этом не нужны.

## Сборка из исходников

Нужны **JDK 17** и **Maven 3.9.4 или новее** (проверяется сборкой; Maven wrapper не входит в комплект).

```powershell
$env:JAVA_HOME = '<путь-к-jdk-17>'
mvn clean verify
```

Готовый p2-репозиторий появится в
`repositories/ru.jimmo.edt.sonarq.repository/target/repository/` (плюс zip-архив того же
содержимого).

## Известные ограничения

- С сервера отдаётся **не более 10 000 замечаний** (ограничение Web API SonarQube); при
  превышении сузьте фильтры на стороне сервера.
- Ветки поддерживаются только на коммерческих редакциях SonarQube; на Community Edition
  показывается единственная основная ветка.
- В локальном режиме анализируется каталог `src/` проекта (типовая раскладка конфигурации
  1С), а при его отсутствии — корень проекта.
- Классификация причин для рекомендованного профиля («Проверки BSL») — заведомо неполный
  первый срез, встроенный в плагин; уточняется сообществом через Pull Request (см.
  [issue #3](https://github.com/Jimmo910/edt-sonarq-plugin/issues/3)).
- Фильтр подсистем сопоставляется по имени подсистемы (так работает сам BSL Language Server),
  поэтому одноимённые подсистемы под разными родителями фильтруются вместе.
